# 审查本地改动

> 审查本地代码仓库中所有未提交的改动(working tree diff / staged diff / untracked files)，帮助用户决定"哪些该提交、哪些该丢弃、哪些需要进一步处理"。最终决策权始终在用户手里。

## 核心红线(不可违反)

1. **禁止擅自 `git commit`**。任何时候都不能在用户没有明确说"commit"、"提交"、"帮我commit这些"之类指令前执行提交操作，哪怕用户已经认可了你的分类结果或提交方案。认可方案 ≠ 授权提交，必须等待独立的、明确的提交指令。
2. **禁止擅自 `git push`**。同理，即使已经commit(无论是用户自己commit还是经明确授权由你commit)，也不能主动push到remote，除非用户单独明确要求push。
3. 如果用户的指令模糊(比如"按你的建议来")，视为对**分类/分组方案**的确认，不能视为对commit/push动作本身的授权。此时应该反问一句"确认要我执行commit吗?"再动手。
4. 上述两条红线不因"看起来改动很小/很安全""用户很着急"等理由放松。

## 整体流程

### 第一步:收集改动信息

不要只看 `git diff` 摘要就下结论，尽量收集足够上下文:

```bash
git status --porcelain=v2 --branch      # 总览:staged/unstaged/untracked
git diff                                 # 未暂存的改动
git diff --cached                        # 已暂存的改动
git status --ignored                     # 确认哪些文件被.gitignore忽略(避免重复提醒)
git diff --stat                          # 快速看改动规模，决定要不要分批读
```

对于 untracked 文件，单独列出(`git status --porcelain` 中 `??` 开头的)，不要遗漏，因为最容易混入临时文件/构建产物。

如果改动量很大，按文件/目录分批读取完整diff，不要因为省事而只看 `--stat` 摘要就分类。

### 第二步:逐项分类

把每一个改动的文件(或同一文件里不同的改动块，如果一个文件里混杂了不同性质的改动)归入以下三类之一:

**A. 有效的 feature / bug fix**
- 实现了明确的功能需求，或修复了明确的bug
- 改动逻辑自洽，没有遗留的调试代码
- 判断依据:改动是否与项目的正常演进逻辑一致；是否有对应的issue/需求/之前的对话上下文支持

**B. 用于验证的临时改动(不应该原样提交，但也不是垃圾，可能需要还原或整理后提交)**
常见特征:
- 临时加的 `print`/`console.log`/`debugger`/`pdb.set_trace()` 等调试语句
- 临时放宽的超时时间、临时写死的测试数据、临时跳过的校验(`if True: # TODO remove`)
- 为了本地验证而改的配置(端口、密钥占位符、mock开关)
- 临时注释掉的代码块用于对比测试
- 改动本身逻辑是对的，但混着上面这些"手脚架"代码

**C. 绝不应该commit的临时文件或打包产物**
常见特征(结合语言/框架识别，不限于此):
- 构建产物:`dist/`、`build/`、`*.class`、`*.o`、`*.pyc`、`__pycache__/`、`target/`(java/rust)、`.next/`、`node_modules/`
- 编辑器/系统临时文件:`.DS_Store`、`*.swp`、`.idea/`、`.vscode/`(除非项目约定要提交)
- 日志和缓存:`*.log`、`.cache/`、`tmp/`、`*.tmp`
- 依赖锁文件是否该提交要看项目惯例(例如`package-lock.json`通常该提交，但`.venv/`不该)
- 明显的密钥/凭证文件:`.env`(含真实值)、`*.pem`、`id_rsa`等 —— 这类要特别标红提示用户，可能是敏感信息泄露风险
- 已经在`.gitignore`里但仍处于untracked状态、被用户`git add -f`强行加入的文件，要特别提醒

### 第三步:处理不确定的项

对于无法直接归类的改动，不要猜，主动查证:

- **查文档**:项目里有没有 `CONTRIBUTING.md`、`README`、`.gitignore`、code style文档等说明规范
- **回溯历史**:
  ```bash
  git log --oneline -- <file>       # 这个文件过去是怎么提交的
  git log -p -1 -- <file>           # 最近一次相关提交长什么样
  git blame <file>                  # 这段代码是谁、什么时候引入的，提交信息是什么
  ```
  用来判断:这类改动过去是被提交过还是一直被排除；这个文件是否历史上就是"不提交"的类型
- 如果查完文档和历史仍不确定，**在最终报告里明确列为"不确定项"并给出你的倾向性建议和理由**，不要替用户悄悄归类到某一类里蒙混过关

### 第四步:输出分类汇总报告

以下面结构向用户汇报(可以根据实际情况增减，但分类结构要保留):

```markdown
## 改动审查汇总

### ✅ 有效的 feature / bug fix
- `path/to/file.py`:实现了 XXX 功能 / 修复了 XXX bug
- ...

### ⚠️ 用于验证的临时改动(建议清理后再提交，或单独处理)
- `path/to/file.py:23`:临时的 print 调试语句
- `config.yaml`:临时放宽了超时时间(60s → 600s)，用于本地压测
- ...

### 🚫 绝不应提交的临时文件/产物
- `dist/bundle.js`(构建产物)
- `.env`(⚠️ 含疑似真实密钥，请确认是否已泄露/是否需要清理git历史)
- `debug.log`
- ...

### ❓ 不确定项(已查阅文档/历史，仍需你确认)
- `path/to/file.ts`:不确定这是有意的重构还是临时试验，历史上该文件改动频率较低，上次改动是 3 个月前的 feature 提交
```

### 第五步:给出提交方案建议

基于以上分类，提出**分组提交**建议，例如:

```markdown
## 建议的提交方案

1. **提交 1**(feature):`src/api/user.py`, `src/api/routes.py` —— 建议commit message: "..."
2. **提交 2**(bug fix):`src/utils/parser.py` —— 建议commit message: "..."
3. **需要清理后再提交**:`src/main.py` 中第 45-52 行的调试代码建议先删除，再并入提交1
4. **建议 `git rm --cached` 或加入 `.gitignore`**:`dist/`, `.env`, `debug.log`
5. **不确定，等待你的决定**:`path/to/file.ts`
```

如果同一个文件里混有A类和B/C类改动(例如一个正常函数改动里混了一行debug print)，要指出具体建议:是"整体提交"还是"部分暂存(`git add -p`)去掉调试行后再提交"。

### 第六步:等待用户决策，严禁自作主张

- 汇报完分类和提交方案后，**明确询问用户**:"以上分组你觉得OK吗?需要调整吗?" 以及(如果讨论到执行层面)"需要我执行commit吗?"
- 只有在用户明确说类似"commit吧""帮我提交第1组""push上去"这类指令后，才能执行对应的 `git commit` / `git push`
- 如果用户只是说"看起来不错"、"就按这个来"这种偏认可分类方案的话，应视为对**分组方案**的确认，仍需追问一句是否现在就执行commit，而不是直接开始commit
- 执行commit时，只commit用户明确同意的文件/分组，不要顺手把"不确定项"也带上
- 执行push前，同样需要独立确认，哪怕commit是刚被用户要求做的

## 小贴士

- 优先使用 `git add -p` / `git restore -p` 之类命令处理"一个文件里混杂多类改动"的情况，而不是整文件二选一
- 如果发现疑似密钥/凭证被添加或已经提交过，要明确提醒用户这是安全风险，而不是简单归为"垃圾文件"了事
- 涉及 `.gitignore` 调整建议时，可以顺手给出建议追加的规则，但同样只在用户同意后才去修改 `.gitignore` 文件本身
